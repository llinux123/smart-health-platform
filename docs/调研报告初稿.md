## 1. 项目背景

### 1.1 行业背景

随着"健康中国 2030"战略的深入推进和互联网医疗政策的持续放开，线上问诊、电子处方、智能分诊等数字化医疗服务已成为医疗信息化建设的核心方向。传统医疗体系面临挂号难、就诊效率低、医疗资源分布不均等结构性问题，亟需通过技术手段实现医疗服务的线上化、智能化。

与此同时，大语言模型（LLM）与检索增强生成（RAG）技术的成熟，为医疗 AI 辅助问诊提供了新的技术路径——通过结合医学知识库的向量检索与大模型的自然语言理解能力，可以在患者端提供初步的症状分析与健康咨询，有效缓解医疗资源紧张的问题。

### 1.2 项目定位

本项目为**智慧医疗与大健康管理平台**（Smart Health Platform），定位为覆盖互联网医院核心业务链路的综合性医疗服务平台。平台以"患者上传图片 → AI 问诊 → 挂号 → 处方 → 库存扣减"为核心链路，整合用户认证、高并发挂号秒杀、AI 智能问诊、电子处方管理四大业务域，旨在提供一个端到端可演示、架构可扩展的完整解决方案。

### 1.3 技术背景

- **运行架构**：Maven 多模块单体架构（单 JVM），按业务域严格分包，保留微服务边界
- **核心中间件**：MySQL 8.x（持久化）、Redis 7.x（缓存与分布式锁）、RabbitMQ 3.x（异步消息）、Elasticsearch 8.x（RAG 检索）
- **AI 能力层**：Spring AI 对接 OpenAI Compatible API，支持多模态图片分析与 SSE 流式对话
- **前端方案**：Vue + Element 管理模板/低代码方案，实现患者端 H5 页面

## 2. 当前业务痛点（Problem Statement）

基于 PRD 文档中定义的问题域与 User Stories，当前业务面临以下核心痛点：

### 2.1 业务链路断裂，缺乏端到端整合

现有互联网医院系统往往各模块独立运作——挂号系统、问诊系统、处方系统之间缺乏统一的数据流转与业务衔接。患者从"发现症状 → AI 初步咨询 → 挂号预约 → 医生诊断 → 开具处方 → 药品配送"的完整链路无法在单一平台内闭环完成，导致用户体验碎片化、运营效率低下。

### 2.2 高并发场景下的号源超卖风险

专家号源稀缺是医疗场景的典型问题。在集中放号时段，大量患者同时抢号会导致数据库层面的激烈竞争。传统基于数据库直接扣减库存的方案在高并发下极易出现超卖现象，严重影响系统公信力和用户体验。

### 2.3 AI 问诊的"幻觉"问题与可信度不足

通用大语言模型在医疗领域的回答存在"幻觉"风险——即生成看似合理但缺乏医学依据的内容。在医疗健康场景下，此类错误信息可能造成严重后果。如何将 AI 回答锚定在真实医学文献上，实现可溯源、可验证的智能问诊，是核心挑战。

### 2.4 处方流转与库存管理脱节

传统电子处方系统与药房库存管理之间缺乏实时联动。处方开具后无法即时扣减对应院区的药品库存，导致库存数据滞后、药品超卖或患者取药时缺货等问题。

### 2.5 系统扩展性不足，技术债务累积

许多医疗信息化系统采用紧耦合架构，业务模块间边界模糊。当需要扩展新功能（如从单体拆分为微服务、引入新的中间件）时，改动范围大、风险高、周期长，严重制约了系统的演进能力。

## 3. 整体解决方案概述

### 3.1 架构方案

采用 **Maven 多模块单体架构**，按业务域划分为六个模块，所有模块在同一 JVM 中运行：

| 模块名称 | 职责边界 |
|---------|---------|
| `smart-health-common` | 通用工具、统一响应体（`Result<T>`）、全局异常处理、常量定义、安全上下文 |
| `smart-health-user` | 患者注册/登录、JWT 签发与校验、Spring Security 认证配置 |
| `smart-health-registration` | 医生排班管理、秒杀抢号（Redis 预扣 + Redisson 分布式锁）、RabbitMQ 异步出单 |
| `smart-health-consultation` | 多模态图片分析、RAG 检索增强问诊、SSE 流式对话、会话上下文管理 |
| `smart-health-prescription` | 电子处方开具、药房库存扣减、药师审核、处方 PDF 存根生成 |
| `smart-health-app` | Spring Boot 启动模块，聚合所有业务模块 |

**关键设计原则**：模块间通过 Java `interface` 层通信，配置文件外部化，数据库表按生产标准设计。后续拆分为微服务时，业务代码零改动。

### 3.2 核心链路设计

```
患者上传患处照片/报告
       ↓
多模态大模型识别 → 生成症状自查草稿
       ↓
基于草稿的 RAG 问诊（ES 双路检索：文本 + 向量）
       ↓
SSE 流式输出 AI 回答（附带引用来源）
       ↓
患者选择医生排班 → 秒杀抢号
       ↓
Redis 预扣库存 + Redisson 分布式锁
       ↓
RabbitMQ 异步生成挂号订单
       ↓
医生查看问诊记录 → 开具电子处方
       ↓
扣减对应院区药房库存 → 处方审核 → PDF 存根
```

### 3.3 基础设施方案

通过 Docker Compose 一键编排所有中间件（MySQL、Redis、RabbitMQ、Elasticsearch），开发环境搭建不超过 5 分钟。数据库初始化脚本自动建表，ES 索引自动创建并初始化医学知识库文档。

## 4. 核心商业痛点推导

基于 PRD 中 29 条 User Stories 的系统分析，推导出以下核心商业痛点：

### 4.1 互联网医院全链路数字化需求迫切

**来源 Stories**：US#17（多轮追问）、US#22（医生查看问诊记录）、US#23（开具处方）、US#28（端到端 H5 流程）

当前市场上缺乏能够将"AI 辅助问诊 → 挂号 → 处方 → 库存"完整链路在单一平台内闭环的解决方案。医疗服务的各环节分散在不同系统中，数据孤岛严重。本项目通过统一平台整合全链路，消除信息断层。

### 4.2 高并发医疗资源分配的公平性与一致性

**来源 Stories**：US#10（专家号秒杀）、US#11（防超卖）、US#12（异步出单）、US#14（订单唯一性）

热门专家的号源分配是互联网医院的核心矛盾。需要在数百甚至数千用户同时抢号的场景下，保证：
- 库存绝对不超卖（数据一致性）
- 响应时间在可接受范围内（性能要求）
- 订单全局唯一可追溯（合规要求）

这要求系统具备金融级的高并发处理能力。

### 4.3 AI 医疗咨询的可信度与合规性

**来源 Stories**：US#15（上传图片）、US#18（RAG 知识库检索）、US#20（引用来源可溯源）、US#21（会话上下文）

医疗 AI 的回答必须基于真实医学文献，而非模型的"臆测"。通过 RAG 技术将 AI 回答锚定在 Elasticsearch 中的医学知识库（指南、药品说明书等），并附带引用来源，是解决 AI 医疗合规性问题的关键技术路径。

### 4.4 处方与库存的实时联动

**来源 Stories**：US#24（库存扣减）、US#25（处方唯一编码 + PDF）、US#26（药师审核）

电子处方的开具必须与药房库存实时联动，确保"开得出、拿得到"。同时处方需要全国唯一编码和 PDF 存根，满足医疗合规与存档要求。药师审核环节保障合理用药。

### 7.2 FR-MOD-01：基础设施与环境管理

#### FR-0101 中间件一键编排

- **需求标识**：FR-0101
- **优先级**：P0（关键）
- **描述**：系统提供 Docker Compose 编排文件，一键启动 MySQL 8.x、Redis 7.x、RabbitMQ 3.x（含 Management 插件）、Elasticsearch 8.x 四个中间件实例。
- **输入**：`docker-compose.yml` 配置文件
- **处理**：
  1. 拉取指定版本的容器镜像
  2. 按配置映射端口（3306/6379/5672/15672/9200）
  3. 挂载数据卷实现数据持久化
  4. 自动执行数据库初始化脚本（`init.sql`）
- **输出**：四个中间件实例运行就绪，端口可访问
- **前置条件**：宿主机已安装 Docker 与 Docker Compose
- **后置条件**：所有中间件处于 Running 状态
- **来源**：US#1

#### FR-0102 统一 API 响应格式

- **需求标识**：FR-0102
- **优先级**：P0（关键）
- **描述**：所有 REST API 接口返回统一的 JSON 响应结构 `{code, message, data}`。
- **输入**：任意 API 请求
- **处理**：通过 `Result<T>` 泛型封装类统一包装返回值
- **输出**：
  ```json
  {
    "code": 200,
    "message": "success",
    "data": { ... }
  }
  ```
- **前置条件**：无
- **后置条件**：调用方可按统一格式解析响应
- **来源**：US#5

#### FR-0103 全局异常处理

- **需求标识**：FR-0103
- **优先级**：P0（关键）
- **描述**：系统通过全局异常处理器捕获所有未处理异常，返回结构化错误信息。
- **输入**：触发异常的 API 请求
- **处理**：
  1. `GlobalExceptionHandler` 拦截异常
  2. 区分业务异常（`BusinessException`）与系统异常
  3. 业务异常返回对应错误码与消息；系统异常返回通用错误信息
- **输出**：结构化错误响应 `{code: 错误码, message: 错误描述, data: null}`
- **前置条件**：无
- **后置条件**：异常不向客户端暴露系统内部堆栈
- **来源**：US#6

#### FR-0104 API 文档自动生成

- **需求标识**：FR-0104
- **优先级**：P1（重要）
- **描述**：集成 SpringDoc + Knife4j，自动生成 OpenAPI 3.0 接口文档，支持通过 Web UI 在线调试。
- **输入**：Controller 层注解与代码结构
- **处理**：扫描 `@RestController`、`@Operation` 等注解，自动生成 API 文档
- **输出**：可通过 `/doc.html` 访问的交互式 API 文档界面
- **前置条件**：应用启动成功
- **后置条件**：所有 API 端点可在 UI 中查看与调试
- **来源**：US#7

### 7.3 FR-MOD-02：用户认证与授权

#### FR-0201 患者注册

- **需求标识**：FR-0201
- **优先级**：P0（关键）
- **描述**：患者通过手机号注册账号，填写必要信息后完成注册。
- **输入**：`RegisterRequest`（用户名、密码、真实姓名、身份证号、手机号、性别）
- **处理**：
  1. 校验用户名与身份证号唯一性
  2. 密码使用 BCrypt 加密
  3. 写入 `t_patient` 表
- **输出**：注册成功返回统一响应
- **前置条件**：无
- **后置条件**：`t_patient` 表新增一条记录
- **来源**：US#2

### 7.5 FR-MOD-04：AI 智能问诊

#### FR-0401 多模态图片分析

- **需求标识**：FR-0401
- **优先级**：P0（关键）
- **描述**：患者上传患处照片或体检报告截图，系统调用多模态大模型识别并生成症状自查草稿。
- **输入**：Multipart 图片文件（`/api/v1/ai/multimodal/analyze`）
- **处理**：
  1. 接收并保存图片至本地文件系统
  2. 调用多模态大模型 API 分析图片内容
  3. 生成结构化症状自查草稿（症状描述、可能病因、建议科室）
  4. 草稿内容存入 `t_consultation_session.symptom_draft`
- **输出**：`MultimodalAnalyzeResponse`（症状草稿、建议科室、草稿ID）
- **前置条件**：患者已认证（FR-0203）
- **后置条件**：`t_consultation_session` 创建会话记录，`symptom_draft` 字段已填充
- **来源**：US#15, US#16

#### FR-0402 RAG 检索增强问诊

- **需求标识**：FR-0402
- **优先级**：P0（关键）
- **描述**：基于症状草稿，系统从 Elasticsearch 医学知识库中检索相关内容，结合大模型生成有据可依的回答。
- **输入**：`ConsultStreamRequest`（会话ID、用户追问内容、症状草稿ID）
- **处理**：
  1. 从 `t_consultation_session` 加载会话上下文与历史对话
  2. 构造检索查询，调用 Elasticsearch 双路检索：
     - **文本检索**：基于 `ik_max_word` 分词的全文匹配
     - **向量检索**：基于 1536 维 `dense_vector` 的 cosine 相似度匹配
  3. 将检索到的医学知识文档作为上下文注入 Prompt
  4. 调用大模型生成回答，附带引用来源
- **输出**：AI 回答文本（含引用来源列表）
- **前置条件**：症状草稿已生成（FR-0401）；ES 知识库索引 `idx_medical_knowledge` 已初始化
- **后置条件**：对话日志追加至 `chat_log` 字段（JSON 格式）
- **来源**：US#17, US#18, US#20, US#21

#### FR-0403 SSE 流式对话输出

- **需求标识**：FR-0403
- **优先级**：P0（关键）
- **描述**：AI 回答以 Server-Sent Events（SSE）流式方式输出，实现打字机效果。
- **输入**：`ConsultStreamRequest`（同 FR-0402）
- **处理**：
  1. 调用大模型流式 API（`streaming mode`）
  2. 逐 token 接收响应片段
  3. 封装为 `ConsultStreamResponse` 通过 SSE 推送至客户端
  4. 流结束后将会话上下文持久化
- **输出**：SSE 事件流（`text/event-stream`），每个事件包含增量文本片段
- **前置条件**：RAG 检索完成（FR-0402）
- **后置条件**：会话上下文已更新（`chat_log` 字段追加）
- **来源**：US#19, US#21
