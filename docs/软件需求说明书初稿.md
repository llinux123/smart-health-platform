## 1. 引言

### 1.1 目的

本文档旨在详细描述"智慧医疗与大健康管理平台"的软件需求，涵盖功能需求、非功能需求、接口需求及设计约束。本文档作为系统设计、开发、测试及验收的基准文件，面向开发人员、测试人员、项目经理及业务方。

### 1.2 范围

本系统是一个端到端可演示的智慧医疗平台，覆盖互联网医院核心业务链路：

- **患者端**：注册登录 → 上传图片 → AI 问诊 → 挂号抢号 → 查看处方
- **运营端**：排班管理
- **医生端**：查看问诊记录、开具处方（第一阶段仅接口预留）
- **药师端**：处方审核（第一阶段仅接口预留）

系统采用 Maven 多模块单体架构，按业务域严格分包（user / registration / consultation / prescription），所有模块在同一 JVM 中运行，保留微服务边界，后续可零改动拆分为微服务。

## 2. 总体描述

### 2.1 产品视角

本系统为独立运行的智慧医疗平台，集成以下核心能力：

- **用户认证**：基于 Spring Security + JWT 的患者身份认证体系
- **挂号秒杀**：基于 Redis 预扣库存 + Redisson 分布式锁 + RabbitMQ 异步出单的高并发抢号
- **AI 问诊**：基于 Spring AI 的多模态图片分析 + RAG 知识库检索 + SSE 流式对话
- **处方管理**：电子处方开具、药房库存扣减、处方审核、PDF 存根生成
- **前端展示**：Vue + Element 患者端 H5 页面，支持端到端完整流程演示

### 2.2 产品功能概要

| 模块 | 核心功能 |
|------|----------|
| 基础设施与用户认证 | Docker Compose 一键启动中间件、患者注册/登录、JWT 认证、统一响应格式、全局异常处理、API 文档 |
| 挂号预约（高并发秒杀） | 排班管理、号源查询、秒杀抢号、Redis 预扣库存、分布式锁防超卖、MQ 异步出单、订单状态管理 |
| AI 智能问诊 | 多模态图片分析、症状草稿生成、RAG 知识库检索、SSE 流式对话、多轮追问、引用来源溯源 |
| 处方与药品管理 | 电子处方开具、药房库存扣减、处方唯一编码、PDF 存根、处方审核、处方查询 |
| 前端展示 | 患者端 H5 页面，覆盖注册、登录、上传、问诊、挂号、查看处方完整流程 |

### 2.3 用户角色

| 角色 | 说明 |
|------|------|
| 患者 | 平台核心用户，使用注册登录、AI 问诊、挂号抢号、查看处方等功能 |
| 运营人员 | 管理医生排班计划，创建号源 |
| 医生 | 查看患者问诊记录，开具电子处方（第一阶段接口预留） |
| 药师 | 审核处方（通过/驳回）（第一阶段接口预留） |
| 系统管理员 | 负责基础设施部署与中间件管理 |
| 开发者 | 通过 API 文档调试接口，进行二次开发 |

| 组件 | 版本要求 |

| JDK | 17+ |
| Spring Boot | 3.2.x |
| MySQL | 8.0 |
| Redis | 7.x（Alpine） |
| RabbitMQ | 3.x（含 management 插件） |
| Elasticsearch | 8.12.0 |

### 2.5 设计与实现约束

- **架构约束**：采用 Maven 多模块单体架构，按业务域分包，单 JVM 运行，保留微服务边界
- **事务约束**：第一阶段使用本地事务，不引入分布式事务（Seata），但预留扩展点
- **存储约束**：会话日志暂存 MySQL text 字段（JSON 格式），不引入 MongoDB
- **AI 约束**：通过 Spring AI 对接 OpenAI Compatible API，切换模型仅改配置不改代码
- **前端约束**：采用 Vue + Element 模板/低代码方案，仅实现患者端

## 3. 功能需求

### 3.1 基础设施与用户认证模块

#### FR-1.1 Docker Compose 一键启动中间件

- **优先级**：高
- **描述**：系统提供 docker-compose.yml 文件，一键启动 MySQL 8.0、Redis 7.x、RabbitMQ 3.x（含 management 插件）、Elasticsearch 8.12.0 四个中间件容器
- **验收标准**：
  - 执行 `docker-compose up -d` 后所有中间件在 5 分钟内启动完毕
  - MySQL 自动执行建表脚本（sql/init.sql）
  - Redis 开启 AOF 持久化
  - RabbitMQ 管理界面可通过 15672 端口访问
  - Elasticsearch 以单节点模式运行，关闭 xpack 安全认证

#### FR-1.2 患者注册

- **优先级**：高
- **触发**：患者提交注册信息（账号、密码、真实姓名、身份证号、手机号、性别）
- **前置条件**：账号未被注册（username 唯一约束）、身份证号唯一
- **处理逻辑**：
  1. 校验请求参数合法性（JSR-303 校验）
  2. 检查用户名是否已存在
  3. 检查身份证号是否已注册
  4. 密码加密存储（BCrypt）
  5. 写入 t_patient 表
- **输出**：统一格式响应 `{code: 200, message: "注册成功", data: null}`
- **接口**：`POST /api/v1/auth/register`

#### FR-1.3 患者登录

- **优先级**：高
- **触发**：患者提交账号密码
- **处理逻辑**：
  1. 校验请求参数合法性
  2. 通过 Spring Security AuthenticationManager 验证账号密码
  3. 验证通过后签发 JWT Token（包含患者 ID、用户名等信息）
  4. 返回 Token 及用户基本信息
- **输出**：`{code: 200, message: "登录成功", data: {token, patientId, username, realName}}`
- **接口**：`POST /api/v1/auth/login`

#### FR-1.4 JWT Token 认证

- **优先级**：高
- **描述**：所有受保护的 API 接口要求请求头携带 `Authorization: Bearer <token>`
- **处理逻辑**：
  1. JwtAuthenticationFilter 拦截请求，提取 Token
  2. 验证 Token 签名与有效期
  3. 解析 Token 中的患者信息，构建 SecurityContext
  4. Token 无效或过期时返回 401 未认证错误
- **豁免接口**：注册、登录、API 文档相关接口

#### FR-1.5 统一 API 响应格式

- **优先级**：高
- **描述**：所有 API 接口返回统一的 JSON 结构
- **格式定义**：
```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```
- **错误码规范**：
  - 200：成功
  - 400：请求参数错误
  - 401：未认证 / Token 无效
  - 403：权限不足
  - 500：服务器内部错误
  - 自定义业务错误码由 ResultCode 枚举定义

#### FR-1.6 全局异常处理

- **优先级**：高
- **描述**：通过 @RestControllerAdvice 全局异常处理器捕获所有未处理异常
- **处理逻辑**：
  - BusinessException（业务异常）：返回对应业务错误码和消息
  - MethodArgumentNotValidException（参数校验异常）：返回 400 及字段校验错误详情
  - AuthenticationException（认证异常）：返回 401
  - AccessDeniedException（权限异常）：返回 403
  - Exception（未知异常）：返回 500 及通用错误消息
- **输出**：所有异常均返回统一 Result 格式，不暴露堆栈信息

#### FR-1.7 API 文档自动生成

- **优先级**：中
- **描述**：集成 SpringDoc + Knife4j，自动生成 OpenAPI 3.0 接口文档
- **验收标准**：
  - 所有 Controller 接口通过 @Tag、@Operation 注解描述
  - 可通过 `/doc.html` 访问 Knife4j 增强文档界面
  - 支持在线调试所有 API（含 JWT Token 配置）

### 3.2 挂号预约（高并发秒杀）模块

#### FR-2.1 创建医生排班

- **优先级**：高
- **角色**：运营人员
- **触发**：运营人员提交排班信息（医生 ID、科室名称、出诊日期、班次、总号源量、挂号费）
- **处理逻辑**：
  1. 校验请求参数（日期不为过去日期、号源量 > 0、挂号费 >= 0）
  2. 创建排班记录写入 t_doctor_schedule 表
  3. 将号源数据同步至 Redis（key: `schedule:{id}:stock`，value: 可用号源量）
- **接口**：`POST /api/v1/schedule/create`

#### FR-2.2 查询可预约排班列表

- **优先级**：高
- **角色**：患者
- **触发**：患者请求排班列表，可按科室名称、出诊日期筛选
- **处理逻辑**：
  1. 根据筛选条件查询 t_doctor_schedule 表
  2. 返回排班列表（含医生 ID、科室、日期、班次、剩余号源、挂号费）
- **输出**：排班 VO 列表
- **接口**：`GET /api/v1/schedule/list`

#### FR-2.3 秒杀抢号

- **优先级**：高（核心链路）
- **角色**：患者
- **触发**：患者提交抢号请求（排班 ID）
- **前置条件**：患者已登录、排班存在且有余量
- **处理逻辑**：
  1. **Redis 预扣库存**：对 Redis 中对应排班的库存 key 执行 DECR 操作，若结果 < 0 则直接返回"号源已满"，并恢复库存（INCR）
  2. **Redisson 分布式锁**：以 `lock:schedule:{scheduleId}:patient:{patientId}` 为锁 key，防止同一患者并发重复抢号
  3. **重复抢号校验**：检查该患者是否已对该排班创建过订单
  4. **发送 MQ 消息**：将抢号成功事件发送至 RabbitMQ 队列（exchange: `registration.exchange`，routingKey: `registration.seckill`）
  5. **异步出单**：MQ 消费者监听队列，执行数据库号源扣减（UPDATE visible_count）、生成挂号订单（写入 t_registration_order）
  6. **立即响应**：主流程返回"抢号成功，订单生成中"，不等待数据库写入完成
- **防超卖机制**：
  - Redis DECR 原子操作保证库存不超卖
  - Redisson 分布式锁保证同一患者不重复抢号
  - 数据库层 visible_count 乐观锁（version 字段）兜底
- **接口**：`POST /api/v1/registration/seckill`

#### FR-2.4 挂号订单号生成

- **优先级**：高
- **描述**：挂号订单号全局唯一，格式为 `REG_{yyyyMMdd}_{序号}`
- **生成规则**：
  - 前缀：`REG_`
  - 日期：8 位日期字符串（如 `20260629`）
  - 序号：当日内的自增序号（可通过 Redis INCR 生成）
- **示例**：`REG_20260629_00001`

#### FR-2.5 查看挂号订单

- **优先级**：高
- **角色**：患者
- **触发**：患者请求查看自己的挂号订单
- **处理逻辑**：根据患者 ID 查询 t_registration_order 表，按创建时间倒序排列
- **订单状态**：
  - 0：排队中
  - 1：待支付
  - 2：已支付
  - 3：已就诊
  - 4：已退号
- **接口**：`GET /api/v1/registration/orders`（预留）

### 3.3 AI 智能问诊模块

#### FR-3.1 多模态图片分析

- **优先级**：高
- **角色**：患者
- **触发**：患者上传患处照片或体检报告截图
- **处理逻辑**：
  1. 接收上传文件（MultipartFile）及图片类型参数（IMAGE-症状图片 / REPORT-检查报告）
  2. 将图片存储至本地文件系统（后续可扩展为 MinIO/OSS）
  3. 调用多模态大模型 API（通过 Spring AI 抽象层）进行图片识别
  4. 大模型返回结构化症状分析草稿（包含可能的症状、建议检查项目、初步判断等）
  5. 将症状草稿保存至 t_consultation_session 表的 symptom_draft 字段
  6. 创建新的问诊会话记录，返回会话编号及分析结果
- **输出**：`MultimodalAnalyzeResponse`（含 sessionId、症状草稿内容、分析结论）
- **接口**：`POST /api/v1/ai/multimodal/analyze`

#### FR-3.2 RAG 知识库检索

- **优先级**：高
- **描述**：AI 回答基于 Elasticsearch 医学知识库的检索增强生成，减少幻觉
- **检索策略**：双路检索
  - **文本检索**：基于 IK 分词的全文检索，匹配关键词相关文档
  - **向量检索**：基于 dense_vector（1536 维，cosine 相似度）的语义检索
- **知识库索引**：`idx_medical_knowledge`
  - 字段：title、content、source_type（指南/说明书/文献）、embedding（向量）、keywords
  - 数据来源：系统初始化时通过 MedicalKnowledgeInitializer 预置
- **检索编排**：
  1. 将用户问题同时发送至文本检索和向量检索
  2. 合并两路结果，按相关性排序
  3. 将检索到的上下文注入 Prompt，交由大模型生成回答

#### FR-3.3 SSE 流式问诊对话

- **优先级**：高（核心链路）
- **角色**：患者
- **触发**：患者发送问诊消息（可携带 sessionId 进行多轮对话）
- **处理逻辑**：
  1. 接收请求参数（sessionId、消息内容）
  2. 若 sessionId 不为空，从 t_consultation_session 加载历史对话上下文（chat_log 字段，JSON 格式）
  3. 执行 RAG 检索，获取相关医学知识上下文
  4. 构建完整 Prompt（系统提示 + RAG 上下文 + 历史对话 + 当前问题）
  5. 通过 Spring AI 调用大模型，以流式方式接收响应
  6. 通过 SseEmitter 将每个 token 实时推送至前端（打字机效果）
  7. 对话结束后，将本轮对话追加至 chat_log 并更新数据库
- **SSE 事件格式**：
  - `event: message`：正常内容 token
  - `event: reference`：引用来源信息
  - `event: done`：回答完成信号
  - `event: error`：错误信息
- **接口**：`POST /api/v1/ai/consult/stream`（Content-Type: text/event-stream）

#### FR-3.4 AI 回答引用来源

- **优先级**：中
- **描述**：AI 回答附带引用来源，确保回答可溯源、可信
- **来源类型**：临床指南、药品说明书、医学文献等
- **展示方式**：通过 SSE `event: reference` 事件推送引用信息，包含来源标题、类型、相关段落

#### FR-3.5 多轮对话上下文管理

- **优先级**：高
- **描述**：多轮对话上下文保存在会话中，支持 AI 理解上下文连贯追问
- **存储方式**：t_consultation_session.chat_log 字段（longtext，JSON 数组格式）
- **上下文策略**：
  - 每次对话保留完整的 user/assistant 消息对
  - 发送给大模型时携带最近 N 轮对话历史（可配置）
  - 会话通过 session_sn 唯一标识，关联患者 ID

### 3.4 处方与药品管理模块

#### FR-4.1 开具电子处方

- **优先级**：高
- **角色**：医生
- **触发**：医生提交处方信息（患者 ID、诊断结论、药品列表）
- **处理逻辑**：
  1. 校验请求参数（患者存在、药品列表非空、各药品数量 > 0）
  2. 生成处方全国唯一编码（格式可参考处方编号规则）
  3. 在本地事务内执行：
     - 创建处方记录（写入 t_prescription）
     - 创建处方明细（写入 t_prescription_item）
     - 扣减对应院区药房库存（UPDATE t_pharmacy_inventory SET stock = stock - quantity）
  4. 生成电子处方 PDF 存根（预留）
- **事务一致性**：处方创建、明细写入、库存扣减在同一本地事务内完成，任一环节失败则全部回滚
- **接口**：`POST /api/v1/prescription/issue`

#### FR-4.2 药房库存扣减

- **优先级**：高
- **描述**：开处方时实时扣减对应院区药房的药品库存
- **扣减逻辑**：
  - 根据 pharmacy_id + medicine_id 定位库存记录
  - 校验实际库存（stock - lock_stock）>= 扣减数量
  - 执行 `UPDATE t_pharmacy_inventory SET stock = stock - #{quantity} WHERE pharmacy_id = #{pharmacyId} AND medicine_id = #{medicineId} AND stock - lock_stock >= #{quantity}`
  - 库存不足时事务回滚，返回库存不足错误
- **多院区支持**：通过 pharmacy_id 字段区分不同院区药房

#### FR-4.3 处方审核

- **优先级**：中（第一阶段接口预留）
- **角色**：药师
- **触发**：药师对待审核处方进行审核
- **处理逻辑**：
  - 审核通过：更新 audit_status = 1
  - 审核驳回：更新 audit_status = 2，填写驳回原因
- **审核状态**：
  - 0：待审核
  - 1：审核通过
  - 2：驳回

#### FR-4.4 查看处方列表与详情

## 4. 非功能需求

### 4.1 高并发防超卖（NFR-1）

- **需求描述**：挂号秒杀场景下，系统必须保证号源不被超卖
- **实现机制**：
  1. **Redis 预扣库存**：
     - 排班创建时将号源量写入 Redis（key: `schedule:{id}:stock`）
     - 抢号时通过 Redis DECR 原子操作预扣库存
     - 库存不足时（DECR 结果 < 0）立即拒绝并恢复库存
     - 避免数据库直接承受高并发写压力
  2. **Redisson 分布式锁**：
     - 锁粒度：`lock:schedule:{scheduleId}:patient:{patientId}`
     - 保证同一患者对同一排班的并发请求串行化处理
     - 锁超时时间可配置（默认 5 秒），防止死锁
  3. **数据库乐观锁兜底**：
     - t_doctor_schedule 表包含 version 字段
     - 扣减 visible_count 时通过 `WHERE version = #{version}` 保证并发安全
     - 更新失败时重试或报错
  4. **RabbitMQ 异步出单**：
     - 主流程仅操作 Redis + 发送 MQ 消息，不阻塞于数据库写入
     - 消费者异步执行数据库操作，提升系统吞吐量
- **验收标准**：
  - 模拟 1000 并发抢 100 个号源，最终订单数 = 100，无超卖
  - 同一患者并发提交多次，仅生成 1 笔订单

### 4.2 SSE 流式响应性能（NFR-2）

- **需求描述**：AI 问诊采用 SSE 流式输出，保证用户体验流畅
- **性能指标**：
  - 首 token 响应时间 ≤ 3 秒（取决于大模型 API 响应速度）
  - token 推送间隔 ≤ 100ms
  - SSE 连接超时时间可配置（默认 60 秒）
  - 支持并发 SSE 连接数 ≥ 50
- **容错机制**：
  - 大模型 API 调用超时自动降级，返回友好错误提示
  - SSE 连接断开时前端支持自动重连
  - 对话异常时保留已输出的内容

### 4.3 JWT 安全认证（NFR-3）

- **需求描述**：系统采用 JWT 进行身份认证，保证接口安全
- **安全要求**：
  - Token 签名算法：HS256 或 RS256
  - Token 有效期：可配置（默认 24 小时）
  - 密码存储：BCrypt 加密，不可逆
  - 敏感信息（密码等）不存入 Token payload
  - Token 中的患者 ID 从服务端解析，不信任客户端传入
- **防护机制**：
  - 无效 Token 返回 401
  - 过期 Token 返回 401 并提示重新登录
  - 受保护接口无 Token 直接拒绝

### 4.4 系统可扩展性（NFR-4）

- **需求描述**：系统架构保留微服务边界，支持后续平滑扩展
- **扩展点设计**：
  1. **模块边界**：各业务模块通过 Java interface 通信，后续拆微服务时替换为 Feign 远程调用即可
  2. **配置外部化**：数据库、Redis、MQ、ES 连接信息均在 application.yml 中配置，支持切换
  3. **AI 层抽象**：通过 Spring AI 抽象层对接大模型，切换模型仅需修改配置
  4. **存储扩展**：图片存储预留接口，当前本地文件系统，后续可切换 MinIO/OSS
  5. **会话存储**：当前 MySQL text 字段，后续可扩展 MongoDB
  6. **知识图谱**：RAG 当前仅用 ES 检索，后续可扩展 Neo4j 图谱验证
  7. **分布式事务**：当前本地事务，后续可扩展 Seata AT 模式
  8. **流量防护**：后续可扩展 Sentinel 限流降级

### 4.5 数据一致性（NFR-5）

- **需求描述**：核心业务场景保证数据一致性
- **一致性保证**：
  1. **挂号秒杀**：
     - Redis 预扣库存保证最终一致性
     - MQ 消费失败时进行重试（配置重试策略）
     - 数据库乐观锁兜底，防止 Redis 与数据库数据不一致
  2. **处方开具**：
     - 处方创建、明细写入、库存扣减在同一本地事务（@Transactional）内完成
     - 任一环节失败全部回滚
  3. **AI 会话**：
     - 对话结束后异步更新 chat_log，不影响 SSE 流式输出
     - 更新失败不影响用户体验（仅丢失对话历史）

### 4.6 可用性（NFR-6）

- **需求描述**：系统开发环境可快速搭建、稳定运行
- **指标**：
  - Docker Compose 一键启动所有中间件，环境搭建 ≤ 5 分钟
  - 应用启动后自动建表/初始化知识库数据
  - 中间件异常时应用可优雅降级（如 ES 不可用时 AI 问诊返回友好提示）

### 4.7 可测试性（NFR-7）

- **需求描述**：系统支持自动化测试，核心逻辑有测试覆盖
- **测试策略**：
  - REST API 集成测试为主（MockMvc / TestRestClient）
  - 单元测试覆盖核心业务逻辑（库存扣减、分布式锁、RAG 检索编排）
  - 不测试框架本身行为（Spring Security 过滤器链、MyBatis 映射等）
- **测试重点**：
  - 挂号秒杀：并发场景下库存不超卖
  - AI 问诊：SSE 流式响应格式正确、RAG 检索返回相关文档
  - 处方开具：库存扣减与处方创建的本地事务一致性
  - 用户认证：Token 签发、过期、无效 Token 拒绝

### 4.8 可维护性（NFR-8）

- **需求描述**：代码结构清晰，易于维护和扩展
- **要求**：
  - 按业务域分包（user / registration / consultation / prescription）
  - 统一日志格式，关键操作记录日志（抢号、处方开具等）
  - 使用 Logback 文件输出日志
  - 代码注释清晰，关键逻辑有 JavaDoc 说明
