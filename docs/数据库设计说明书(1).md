## 1. 引言

### 1.1 编写目的

本文档描述"智慧医疗与大健康管理平台"的数据库设计，包括 MySQL 关系型数据库的表结构、索引策略、存储引擎选型，以及 Elasticsearch 搜索引擎的索引映射设计。本文档面向开发人员、DBA 及运维人员，作为数据库开发、维护和性能优化的参考依据。

### 1.2 设计原则

- **范式化与反范式化平衡**：核心业务表满足 3NF，适当引入冗余字段（如 doctorName）减少关联查询
- **命名规范**：表名 `t_` 前缀 + 业务含义 snake_case，字段名 snake_case，索引名 `idx_`/`uk_` 前缀
- **字符集**：统一 `utf8mb4` + `utf8mb4_unicode_ci`，支持完整 Unicode（含 emoji）
- **存储引擎**：统一 `InnoDB`，支持事务、行级锁、外键约束（逻辑外键）
- **兼容性**：70% 以上字段含 `COMMENT` 注释，便于自动生成代码和文档

### 1.3 数据量预估

| 表名 | 预估记录数 | 增长频率 | 说明 |
|------|-----------|---------|------|
| t_patient | 数百～数千 | 低 | 演示/测试场景 |
| t_doctor | 数十 | 静态 | 预置数据 |
| t_doctor_schedule | 数百～数千 | 中 | 每日排班 |
| t_registration_order | 数千 | 中（秒杀时集中） | 抢号订单 |
| t_prescription | 数百 | 低 | 演示数据 |
| t_prescription_item | 处方数*平均药品数 | 低 | 处方明细 |
| t_pharmacy_inventory | 数十～数百 | 低 | 预置数据 |
| t_consultation_session | 数百 | 中 | AI问诊会话 |
| t_consultation_turn | 会话数*平均轮次 | 中 | 对话明细 |
| t_consultation_rating | 会话数 | 低 | 问诊评分 |
| t_staff | 数十 | 低 | 员工（医生/药师/运维） |

## 3. 表结构详述

### 3.1 `t_patient` — 患者用户表

| 字段名 | 类型 | 空 | 默认值 | 说明 |
|--------|------|----|--------|------|
| id | bigint(20) | NO | AUTO_INCREMENT | 患者ID（PK） |
| username | varchar(50) | NO | - | 登录账号（UK） |
| password | varchar(100) | NO | - | BCrypt加密密码 |
| real_name | varchar(50) | NO | - | 真实姓名 |
| id_card | varchar(18) | NO | - | 身份证号（UK） |
| phone | varchar(11) | NO | - | 手机号（UK） |
| gender | tinyint(1) | YES | 0 | 0:未知 1:男 2:女 |
| is_deleted | tinyint(1) | NO | 0 | 是否已软删除 |
| deleted_at | datetime | YES | NULL | 软删除时间 |
| create_time | datetime | NO | CURRENT_TIMESTAMP | 注册时间 |

**索引**：
- PK: `id`
- UK: `uk_username` (username)
- UK: `uk_id_card` (id_card)
- UK: `uk_phone` (phone)

**DDL**：
```sql
CREATE TABLE `t_patient` (
  `id` bigint(20) NOT NULL AUTO_INCREMENT COMMENT '患者ID',
  `username` varchar(50) NOT NULL COMMENT '登录账号',
  `password` varchar(100) NOT NULL COMMENT '加密密码',
  `real_name` varchar(50) NOT NULL COMMENT '真实姓名',
  `id_card` varchar(18) NOT NULL COMMENT '身份证号',
  `phone` varchar(11) NOT NULL COMMENT '手机号',
  `gender` tinyint(1) DEFAULT '0' COMMENT '性别 (0:未知 1:男 2:女)',
  `is_deleted` tinyint(1) NOT NULL DEFAULT 0 COMMENT '是否已软删除',
  `deleted_at` datetime DEFAULT NULL COMMENT '软删除时间',
  `create_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '注册时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_username` (`username`),
  UNIQUE KEY `uk_id_card` (`id_card`),
  UNIQUE KEY `uk_phone` (`phone`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='患者用户表';
```

### 3.3 `t_doctor_schedule` — 医生排班号源表

| 字段名 | 类型 | 空 | 默认值 | 说明 |
|--------|------|----|--------|------|
| id | bigint(20) | NO | AUTO_INCREMENT | 排班ID（PK） |
| doctor_id | bigint(20) | NO | - | 医生ID（FK → t_doctor） |
| dept_name | varchar(50) | NO | - | 科室名称 |
| work_date | date | NO | - | 出诊日期 |
| shift | tinyint(1) | NO | 1 | 1:上午 2:下午 |
| total_count | int(11) | NO | 0 | 总号源量 |
| visible_count | int(11) | NO | 0 | 剩余可抢号源量 |
| price | decimal(10,2) | NO | 0.00 | 挂号费 |
| version | int(11) | NO | 0 | 乐观锁版本号 |

**索引**：
- PK: `id`
- KEY: `idx_doctor_date` (doctor_id, work_date)

**DDL**：
```sql
CREATE TABLE `t_doctor_schedule` (
  `id` bigint(20) NOT NULL AUTO_INCREMENT COMMENT '排班ID',
  `doctor_id` bigint(20) NOT NULL COMMENT '医生ID',
  `dept_name` varchar(50) NOT NULL COMMENT '科室名称',
  `work_date` date NOT NULL COMMENT '出诊日期',
  `shift` tinyint(1) NOT NULL DEFAULT '1' COMMENT '班次 (1:上午 2:下午)',
  `total_count` int(11) NOT NULL DEFAULT '0' COMMENT '总号源量',
  `visible_count` int(11) NOT NULL DEFAULT '0' COMMENT '剩余可抢号源量',
  `price` decimal(10,2) NOT NULL DEFAULT '0.00' COMMENT '挂号费',
  `version` int(11) NOT NULL DEFAULT '0' COMMENT '乐观锁版本号',
  PRIMARY KEY (`id`),
  KEY `idx_doctor_date` (`doctor_id`,`work_date`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='医生排班号源表';
```

**设计说明**：
- `version` 字段用于乐观锁，防止 MQ 消费者并发更新 visible_count 时出现超卖
- `doctor_id` + `work_date` 联合索引加速按医生+日期的排班查询

### 3.4 `t_registration_order` — 挂号订单表

| 字段名 | 类型 | 空 | 默认值 | 说明 |
|--------|------|----|--------|------|

| patient_id | bigint(20) | NO | - | 患者ID |
| schedule_id | bigint(20) | NO | - | 排班ID |
| sequence_number | int(11) | YES | NULL | 就诊呼叫序号 |
| amount | decimal(10,2) | NO | 0.00 | 支付金额 |
| status | tinyint(1) | NO | 0 | 0:排队中 1:待支付 2:已支付 3:已就诊 4:已退号 |
| pay_time | datetime | YES | NULL | 支付时间 |
| create_time | datetime | NO | CURRENT_TIMESTAMP | 创建时间 |

**索引**：
- PK: `id`
- UK: `uk_order_sn` (order_sn)

**DDL**：
```sql
CREATE TABLE `t_registration_order` (
  `id` bigint(20) NOT NULL AUTO_INCREMENT COMMENT '挂号单ID',
  `order_sn` varchar(64) NOT NULL COMMENT '订单流水号(全局唯一)',
  `patient_id` bigint(20) NOT NULL COMMENT '患者ID',
  `schedule_id` bigint(20) NOT NULL COMMENT '排班ID',
  `sequence_number` int(11) DEFAULT NULL COMMENT '就诊呼叫序号',
  `amount` decimal(10,2) NOT NULL DEFAULT '0.00' COMMENT '支付金额',
  `status` tinyint(1) NOT NULL DEFAULT '0' COMMENT '状态 (0:排队中 1:待支付 2:已支付 3:已就诊 4:已退号)',
  `pay_time` datetime DEFAULT NULL COMMENT '支付时间',
  `create_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_order_sn` (`order_sn`),
  KEY `idx_patient_id` (`patient_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='挂号订单表';
```

**设计说明**：
- `order_sn` 格式：`REG_{yyyyMMdd}_{6位序号}`，通过 Redis INCR 生成
- `status` 状态机：0(排队中) → 1(待支付) → 2(已支付) → 3(已就诊)，4(已退号) 从任意状态转入
- `patient_id` 索引加速患者订单查询

### 3.5 `t_prescription` — 电子处方表

| 字段名 | 类型 | 空 | 默认值 | 说明 |

| audit_comments | varchar(500) | YES | NULL | 审核意见 |

**索引**：
- PK: `id`
- UK: `uk_pres_sn` (prescription_sn)
- KEY: `idx_patient_doc` (patient_id, doctor_id)

**DDL**：
```sql
CREATE TABLE `t_prescription` (
  `id` bigint(20) NOT NULL AUTO_INCREMENT COMMENT '处方ID',
  `prescription_sn` varchar(64) NOT NULL COMMENT '处方全国唯一编码',
  `patient_id` bigint(20) NOT NULL COMMENT '患者ID',
  `doctor_id` bigint(20) NOT NULL COMMENT '开具医生ID',
  `diagnosis` varchar(500) NOT NULL COMMENT '临床诊断结论',
  `pdf_url` varchar(255) DEFAULT NULL COMMENT '电子处方PDF存根路径',
  `audit_status` tinyint(1) NOT NULL DEFAULT '0' COMMENT '药师审核状态 (0:待审核 1:审核通过 2:驳回)',
  `pharmacist_id` bigint(20) DEFAULT NULL COMMENT '审核药师ID',
  `audit_comments` varchar(500) DEFAULT NULL COMMENT '审核意见',
  `audit_time` datetime DEFAULT NULL COMMENT '审核时间',
  `status` tinyint(1) NOT NULL DEFAULT '0' COMMENT '流转状态 (0:未配药 1:配药中 2:已发药)',
  `create_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '开具时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_pres_sn` (`prescription_sn`),
  KEY `idx_patient_doc` (`patient_id`,`doctor_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='电子处方表';
```

**设计说明**：
- `prescription_sn` 格式：`RX_{yyyyMMdd}_{hospitalId}_{6位序号}`
- `audit_status` + `pharmacist_id` + `audit_comments` + `audit_time` 形成完整审核链路

## 7. 数据一致性方案

### 7.1 一致性矩阵

| 业务场景 | 一致性要求 | 实现机制 |
|---------|-----------|---------|
| 挂号秒杀-库存 | 最终一致 | Redis DECR 预扣 → MQ 异步写库 → DB乐观锁版本号兜底 |
| 挂号秒杀-订单 | 最终一致 | MQ 消费者幂等创建订单（order_sn 唯一索引） |
| 处方开具-库存扣减 | 强一致 | @Transactional 本地事务，条件 UPDATE 保证原子性 |
| 处方审核-驳回恢复 | 强一致 | @Transactional 本地事务，更新状态+恢复库存同事务 |
| AI对话-轮次保存 | 最终一致 | SSE 流结束后异步 INSERT，失败不影响用户 |
| 用户注册 | 强一致 | @Transactional + UNIQUE 索引 |

### 7.2 乐观锁实现（秒杀兜底）

```sql
-- t_doctor_schedule 表包含 version 字段
UPDATE t_doctor_schedule
SET visible_count = visible_count - 1,
    version = version + 1
WHERE id = #{scheduleId}
  AND visible_count > 0
  AND version = #{oldVersion}
```

**说明**：MQ 消费者在创建订单后执行此 UPDATE，如果 version 冲突说明已有其他线程先更新，此时事务回滚，订单进入死信队列待人工处理。

### 7.3 条件库存扣减（处方）

```sql
UPDATE t_pharmacy_inventory
SET stock = stock - #{quantity}
WHERE pharmacy_id = #{pharmacyId}
  AND medicine_id = #{medicineId}
  AND stock - lock_stock >= #{quantity}
```

**说明**：利用 MySQL 行锁 + 条件判断（`stock - lock_stock >= quantity`）在单行 UPDATE 层面保证不超卖，affected rows = 0 时事务整体回滚。

mysql -u root -p smart_health < sql/V2__consultation_management.sql

USE smart_health;
